---
title: Judging Modes
description: Understand the two judging modes — points-based and subjective — and when to use each.
---

Oatmeal supports two distinct judging modes. Choose the right one when setting up your hackathon.

## Overview

| Mode | How it works | Best for |
|------|-------------|----------|
| **Points** | Judges score each submission against weighted criteria (e.g., 1-10) | Large hackathons, objective ranking, detailed feedback |
| **Subjective** | Judges pick favorites for each prize category | Small hackathons, simple voting, opinionated categories |

Set the mode via the [Hackathons API](/docs/sdk/hackathons):

```bash title="Set judging mode"
curl -s -X PATCH "$BASE_URL/api/dashboard/hackathons/$HACKATHON_ID/settings" \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"judgingMode": "points"}' | jq .
```

## Points-Based Judging

Judges evaluate each assigned submission against a set of criteria with numeric scores. Final rankings are calculated by aggregating weighted scores across all judges.

### Setup

1. **Define criteria** — Create judging criteria with names, max scores, and weights. Repeat the call for each criterion:

```bash title="Add a judging criterion"
curl -s -X POST "$BASE_URL/api/dashboard/hackathons/$HACKATHON_ID/judging/criteria" \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"name": "Technical Innovation", "maxScore": 10, "weight": 1.5}' | jq .
```

2. **Add judges** and **auto-assign submissions**

```bash title="Auto-assign submissions to judges"
curl -s -X POST "$BASE_URL/api/dashboard/hackathons/$HACKATHON_ID/judging/auto-assign" \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"submissionsPerJudge": 5}' | jq .
```

3. **Judges score** through the web interface at `/e/{slug}/judge`. Each judge sees their assigned submissions and scores them against the criteria you defined.

### How Scoring Works

For each submission, the total score is calculated as:

```
totalScore = Σ (criteriaScore × criteriaWeight)
```

The weighted score normalizes by judge count so submissions with different numbers of reviewers are compared fairly.

### When to Use Points

- 10+ submissions where objective comparison matters
- Multiple judging criteria with different importance levels
- You want detailed scoring breakdowns per criteria
- Transparency in how rankings were determined

## Subjective Judging

Judges pick their favorite submissions for each prize category. Results are determined by tallying picks across all judges.

### Setup

1. **Define prizes** — Each prize becomes a category judges vote on. Repeat the call for each prize:

```bash title="Add a prize category"
curl -s -X POST "$BASE_URL/api/dashboard/hackathons/$HACKATHON_ID/prizes" \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"name": "Best Overall", "type": "favorite"}' | jq .
```

2. **Judges pick favorites** through the web interface at `/e/{slug}/judge`. Each judge selects their top choice for each prize category.

3. **View aggregated results** as the organizer

```bash title="View pick results"
curl -s "$BASE_URL/api/dashboard/hackathons/$HACKATHON_ID/judging/pick-results" \
  -H "Authorization: Bearer $API_KEY" | jq .
```

### When to Use Subjective

- Fewer than 10 submissions
- Prize categories are opinion-based ("Most Creative", "Crowd Favorite")
- You want a faster judging process
- Rankings don't need numeric justification

## Anonymous Judging

Both modes support anonymous judging. When enabled, team names are hidden from judges so they evaluate submissions purely on merit.

```bash title="Enable anonymous judging"
curl -s -X PATCH "$BASE_URL/api/dashboard/hackathons/$HACKATHON_ID/settings" \
  -H "Authorization: Bearer $API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"anonymousJudging": true}' | jq .
```

## Best Practices

1. **Set up criteria before adding judges** so judges know what to evaluate from the start
2. **Use auto-assign** to distribute submissions evenly and avoid conflicts of interest
3. **Enable anonymous judging** for larger events to reduce bias
4. **Monitor progress** — check the assignments endpoint for completion stats before calculating results
5. **Recalculate as needed** — if you add or remove judges, recalculate results before publishing
