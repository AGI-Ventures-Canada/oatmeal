The Oatmeal platform exposes a REST API. All requests use standard `fetch()` with JSON payloads and API key authentication. For a comprehensive overview of all authentication methods, see the [Authentication](/docs/authentication) guide.

## Get Your API Key

Create an API key from the dashboard at **Settings > API Keys**. Keys start with `sk_live_` and are shown only once.

<Callout type="warn">
Never expose your API key in client-side code. Use environment variables and server-side API routes.
</Callout>

## Environment Variables

Store your API key and base URL in an environment variable:

```bash title=".env.local"
OATMEAL_API_KEY=sk_live_...
OATMEAL_BASE_URL=https://your-app.com
```

## Authentication

All API requests require an `Authorization` header with your API key:

```typescript title="HTTP Header"
Authorization: Bearer sk_live_your_api_key_here
```

## Verify Connection

Test your API key with the `whoami` endpoint:

```typescript title="verify.ts"
const res = await fetch(`${process.env.OATMEAL_BASE_URL}/api/v1/whoami`, {
  headers: {
    "Authorization": `Bearer ${process.env.OATMEAL_API_KEY}`,
  },
})

if (!res.ok) {
  console.error("Authentication failed:", res.status)
  process.exit(1)
}

const data = await res.json()
console.log("Connected as tenant:", data.tenantId)
console.log("Key ID:", data.keyId)
console.log("Scopes:", data.scopes)
```

### Response

```json title="response.json"
{
  "tenantId": "org_abc123",
  "keyId": "key_xyz",
  "scopes": ["hackathons:read", "hackathons:write", "webhooks:read", "webhooks:write"]
}
```

## Error Handling

API errors return JSON with an `error` field. Use standard `fetch()` error handling:

```typescript title="api-helper.ts"
async function apiCall(path: string, options?: RequestInit) {
  const res = await fetch(`${process.env.OATMEAL_BASE_URL}${path}`, {
    ...options,
    headers: {
      "Authorization": `Bearer ${process.env.OATMEAL_API_KEY}`,
      "Content-Type": "application/json",
      ...options?.headers,
    },
  })

  const data = await res.json()

  if (!res.ok) {
    throw new Error(`API error ${res.status}: ${data.error}`)
  }

  return data
}
```

## API Namespaces

The API is split across three namespaces with different authentication requirements:

| Namespace | Auth | Description |
|-----------|------|-------------|
| `/api/public/*` | None or Clerk session | Public discovery, participant actions (registration, submissions, voting) |
| `/api/dashboard/*` | API key or Clerk session | Organizer management (hackathon CRUD, judging, prizes, sponsors) |
| `/api/v1/*` | API key only | Programmatic access for jobs and webhooks |

<Callout type="info">
Most organizer and programmatic endpoints accept API keys. Participant-facing actions (registration, submissions, team invitations) require a Clerk session since they act on behalf of a logged-in user.
</Callout>
